FinOps control plane · GitHub Copilot

Governing AI-credit spend after the token-billing shift.

GitHub Copilot moved to usage-based billing on 1 June 2026. The native controls are coarse: budgets are static, forecasting is absent, and the safest-sounding defaults invite bill shock. This document specifies a custom layer for forecasting, budget balancing, and multi-level budget orchestration mapped to GitHub's own constructs.

Part 1 — How billing actually works now

The unit: AI Credits

  • Usage is metered in GitHub AI Credits; 1 credit = $0.01 USD.
  • A credit is consumed from tokens (input + output + cached) at each model's published rate. Cost = f(model, tokens).
  • Code completions and Next Edit Suggestions are free (no credits).
  • Copilot code review also consumes GitHub Actions minutes — a second meter.
No lower-cost fallback

When credits or budget are exhausted, credit-consuming features are blocked, not downgraded.

The two phases and the pools

Every cycle runs in two phases. The controls behave differently in each — the single most important thing to internalise.

Pool phase shared included credits

Consumption draws from a shared pool pooled at the billing-entity level (100 Business seats = one pool of ~190,000 credits, not 100 buckets). Standard ≈ 1,900 (Business) / 3,900 (Enterprise) per seat; promo ≈ 3,000 / 7,000 for existing customers 1 Jun–1 Sep 2026. Resets monthly at 00:00:00 UTC, no carryover. Adding seats grows the pool immediately; removing doesn't shrink it until next cycle.

The cliff

The pool falls ~37% on 1 Sep 2026 when the promo ends — model it as a step-change, not a smooth trend.

Metered phase pay-as-you-go

After the pool is exhausted, usage bills at $0.01/credit — but only if the "AI credit paid usage" policy is enabled. If disabled, pool exhaustion simply blocks (a hard, free ceiling).

Individual Pro/Pro+/Max plans use a separate base + flex allowance model; irrelevant to org control except that assigning an org seat auto-cancels a personal plan with proration.

The control surface — two families plus one lever

Three kinds of control. Conflating them is the most common failure mode.

Family A · User-level budgets both phases

Cap each person's total consumption across both phases. Always a hard stop; $0 blocks immediately. Because they count included usage, they can block a user before the pool is exhausted. The sum of ULBs is an implicit overage ceiling. Three scopes, most-specific wins:

ULB scopeApplies toPrecedence
Individual ULBone named userhighest
Cost center ULB (CCULB) API-ONLYevery current/future member of a cost center (one per-user amount)middle
Universal ULBevery licensed user by defaultlowest

Family B · Spending limits metered only

Cap metered charges only, and only after the pool is exhausted. Pool-agnostic.

Spending limitCapsScope
Cost center budgeta team's metered chargesper cost center
Organization budgetan org's metered chargesper organization
Enterprise budgettotal enterprise metered chargesenterprise-wide
Hard stop is OFF by default

For all three spending limits, "Stop usage when budget limit is reached" must be enabled or charges accrue uncapped. Cost centers can be excluded from the enterprise budget to spend only against their own cap.

Lever C · Cost-center included-usage cap pool phase API-ONLY

A separate control that caps how much of the shared pool a cost center can draw before it tips into metered — what makes chargeback boundaries hold in the pool phase (one team can't drain credits another team's licenses funded). Not amount-settable: GitHub auto-computes the limit from the licenses attributed to the cost center and re-derives it as membership changes — you never enter a number. The only knobs are on/off and, at the cap, block-or-overflow. Enabling it doesn't retroactively redistribute the enterprise pool. Model it as enabled: bool + overflow: block|metered with a read-only computed limit — categorically different from budgets/ULBs, which take a dollar amount.

Lowest remaining headroom wins

Evaluation runs ULB first (most-specific applicable) → pool/metered routing → the relevant spending limit. Whichever applicable control has the least capacity blocks first — so ULBs and spending limits must be sized together.

  1. The two most useful balancing levers (CCULB, included-usage cap) are API-only. A native admin can't carve the pool per team or set per-team per-user caps without the API.
  2. Known UI display bug: user-level budgets created in the UI often don't appear in the "Budgets and alerts" list, blocking edit/delete; the workaround is the REST API. Orgs have blocked everyone with an accidental $0 universal ULB fixable only via API.
Design stance: API-first for all writes

GitHub's UI is treated as read-only/backup, not a parallel control surface — which also removes the out-of-band drift problem.

The gaps that justify a custom tool

  1. No per-user × per-model × per-surface attribution in one API (entity-model or user-total, not both).
  2. No native forecasting (run-rate, cycle burn-down, the 1 Sep cliff).
  3. No budget balancing — native budgets are static; nothing redistributes headroom as the cycle progresses. The tool's centrepiece.
  4. Fragmented, partly API-only, buggy administration — no single reliable surface to run all controls.
  5. Reporting latency, no real-time push; coarse alerting.

Budget management (CRUD)

GET/POST /{enterprises|organizations}/{id}/settings/billing/budgets, plus GET/PATCH/DELETE /budgets/{id} and per-user state for a universal-ULB budget.

Budget object

  • budget_type — ProductPricing · SkuPricing · BundlePricing.
  • budget_product_sku — e.g. ai_credits, actions.
  • budget_scope — enterprise/organization/cost_center/repository (spending limits); user-level (universal/individual ULBs); multi_user_cost_center (the CCULB, API-only).
  • budget_amount (USD) · prevent_further_usage (hard-stop) · budget_alerting.
CCULB call shape

{ budget_amount, prevent_further_usage:true, budget_scope:"multi_user_cost_center", budget_entity_name:"<cost center>", budget_type:"BundlePricing", budget_product_sku:"ai_credits" }

Cost centers & included-usage caps

/enterprises/{id}/settings/billing/cost-centers — GET (list, ?state=active), POST, GET/DELETE /{ccid}, POST/DELETE /{ccid}/resource (users, orgs, repos, enterprise teams). Enterprise-team assignment keeps membership in sync via IdP/SCIM.

Included-usage cap (Lever C): toggled on the cost center via the create/edit API (e.g. ai_credit_pool_enabled) with an overflow choice. No amount field — the limit is auto-computed from attributed licenses and moves with membership, so the shape is enabled: bool + overflow: block|metered, not a settable value. API-ONLY

Part 3 — Product requirements

Problem & goals

Token billing removed the flat-rate ceiling; native controls are static, fragmented, partly API-only, and default to no hard stop. We need one API-first plane that administers every control, forecasts, and — the centrepiece — auto-balances in both phases.

G1
Administer every control
All ULBs, spending limits, included-usage caps, hard-stop flags, cost-center lifecycle. API-first writes.
G2
Forecast
Cycle burn-down and metered spend at every scope, incl. the 1 Sep cliff.
G3
Auto-balance · pool
Redistribute unconsumed shared-pool access to blocked / ≥95% entities before month-end, without tipping into metered.
G4
Auto-balance · metered
Redistribute unused enterprise/org spending headroom to near-capped cost centers or individuals.
G5
Attribution
Chargeback by DEWR division/branch/project via cost centers.
G6
Alerting · audit · RBAC
Custom routing, immutable audit, least-privilege tokens.

Non-goals

  • Not a traffic proxy (GitHub meters server-side); not the billing system of record; not per-request model routing.
  • Does not expand the pool (only license changes do) — it redistributes access.

Personas

Engineering lead
Owns a cost center; wants per-member burn, headroom, alerts.
Enterprise admin

Read-only headroom + projected block date.

Functional requirements

Administration (G1)

  • FR1. API-first CRUD of every control: universal/CCULB/individual ULBs; enterprise/org/cost-center budgets; included-usage caps (block/overflow); hard-stop flags; cost-center lifecycle + membership + enterprise-team resources.
  • FR2. Surface the two API-only controls as first-class UI, badged "API-only".
  • FR3. Reconcile the ULB display bug: work from the API's authoritative list; detect orphaned/hidden budgets; offer repair the UI can't.
  • FR4. Validate on write: enterprise-cap-too-low; multi-org-licensed users; $0 ULBs; enforce prevent_further_usage=true for hard caps.

Forecasting (G2)

  • FR5. Per-entity run-rate → exhaustion date + metered $; weekday seasonality; P50/P90; runway; backtest (MAPE); the 1 Sep step-change.

Auto-balancing · pool phase (G3)

  • FR6–10. Detect trigger (near cycle end + pool underutilised + entities blocked or ≥95%) → size a funding envelope → resolve each entity's binding constraint and branch on type: ULB-bound → grant the most-specific ULB (individual override, else CCULB); cap-bound → no grantable delta, relax only (disable cap / overflow / re-attribute licenses) → simulate (unblocks, projected utilisation, metered-tip probability) before apply. Genuine pool-phase redistribution is ULBs only.

Auto-balancing · metered phase (G4)

  • FR11–14. Detect trigger (metered phase + entity ≥95% + higher-scope headroom) → envelope from remaining enterprise/org budget → resolve the binding budget (lowest-headroom-wins) and raise that one → simulate with the bill delta.

Grant lifecycle (both)

  • FR15. Budgets persist across cycles (only the pool resets). Every auto-grant is time-boxed with a revert/re-baseline-at-rollover policy — preventing ceiling creep.
  • FR16. Guardrails: max grant % / absolute, floors, mandatory reserve buffer, approval gates, full reversibility.

Alerting · audit · RBAC (G6)

  • FR17–19. Custom thresholds + anomaly detection + per-cost-center routing; immutable audit (actor, trigger, envelope, binding constraint, before/after, forecast basis); least-privilege tokens.

Simulation & verification no live credentials yet

  • FR20–22. A full simulation mode requiring no PAT, backed by an MSW mock of the GitHub API — models pooled draw, two-phase routing, ULB precedence, lowest-headroom-wins, the non-settable included-usage cap, the API-only CCULB, and the mutations so simulate-before-apply and both rebalancers run end to end. Shipped as a toggleable offline/demo mode. Same MSW handlers back simulation, e2e, and unit tests (can't drift); stateless — reset to fixtures each run. Simulated actions never reach real GitHub; the mode is unmistakable in the UI.
  • FR23. Verification gate (every change): Playwright e2e (headless, MSW-backed) is the automated gate; Chrome MCP verification of the running app is the interactive confirmation. A change is done only when both pass.

Forecasting

  • Blended trailing burn (trailing-7 weighted + cycle-to-date) with weekday seasonality → exhaustion date + metered $.
  • Allowance is a function of cycle month + existing-customer status, so any forecast crossing 1 Sep uses standard allowances — surfaced as a step-change.
  • P50/P90 from daily-burn variance; nightly backtest (MAPE); latest day provisional (settling window).

Auto-balancing engine

Both rebalancers share one pattern: detect trigger → size a funding envelope → resolve each at-risk entity's binding constraint → allocate greedily by priority → simulate → apply the most-specific lever → track as a grant → revert at rollover. Greedy allocation against a hard envelope is sufficient and explainable; the envelope guarantees we never over-commit.

A · Pool-phase rebalancer — "use it or lose it"

  • Why: unused pool credits are forfeited at reset while some users/cost centers are blocked or ≥95%. Raising a ULB doesn't grow the pool — it raises a ceiling so that user can draw the unconsumed shared slack.
  • Envelope: remaining_pool − reserve − Σ projected(non-at-risk to cycle end).
  • Redistribution levers (the only ones that hand out slack): individual ULB override (surgical — precedence over CCULB, unblocks one user without raising the team) → CCULB (blunt — lifts every member's ceiling uniformly). Cost-center budgets are metered-only and the included-usage cap doesn't grant, so ULBs are the entire redistribution toolkit here.
  • Cap-bound = a different branch (relax, don't redistribute): a team blocked by its included-usage cap while the pool has slack can't be helped by any ULB raise (lowest-headroom-wins; the cap binds). Only moves: disable the cap, overflow → metered, or attribute more licenses. The resolver must distinguish cap-bound (relax-only) from ULB-bound (redistributable).
  • Safety: never let raised ceilings exceed remaining pool + allowed metered; simulate metered-tip probability; approve.
Scenario

Day 26/30, pool 68% consumed (projected 82% → ~18% forfeit), 6 blocked and 11 at ≥95%. Envelope = the ~18% slack minus reserve. Raise individual ULBs for the 17 at-risk users, simulate 96% utilisation with <3% metered-tip risk, apply on approval, tag each as a grant to revert at reset.

B · Metered-phase rebalancer — spending-headroom redistributor

  • Why: after the pool is gone, some cost centers/individuals hit their metered cap while the enterprise/org budget still has large unused headroom.
  • Envelope: remaining_enterprise(or org)_budget − reserve − Σ projected_metered(non-at-risk).
  • Levers: raise the binding budget only (ULB vs cost-center vs org vs enterprise, lowest-headroom). Excluded cost centers are self-funded.
  • Simulate: who stays unblocked, projected total metered $, remaining headroom, and the bill delta — before apply.
Scenario

Enterprise metered budget $8,000, $6,300 unused, 4 days left; cost center Platform at 98% of its $600 cap and 3 individuals near their ULB. Redistribute from the $6,300 envelope: +$250 to Platform and individual-ULB bumps for the 3 users (whichever is binding), simulate the bill delta, apply on approval.

Grant lifecycle — mandatory

Budgets persist across cycles; only the pool resets. Without a lifecycle, end-of-cycle grants become permanent higher ceilings, inflating next month's implicit overage exposure. Every grant is time-boxed with a revert / re-baseline at rollover policy, plus a "grant creep" guard.

Sync engine, auth & security

  • Declarative desired-state (Terraform-style): rebalancers emit desired budgets/caps; engine diffs vs live and applies idempotent POST/PATCH/DELETE; reconciles out-of-band UI edits.
  • API-first for all writes — mandated by the API-only levers + the ULB display bug.
  • Classic PAT with manage_billing:enterprise in Azure Key Vault; separate read/write tokens; write token reachable only by the sync engine's service identity.
  • Government posture: append-only, hash-chained audit; every grant explainable from its recorded trigger + envelope + forecast basis.

Edge cases

  • Lazy universal-ULB records → reconcile against full licensing. Mid-cycle seat add/remove → forecast on effective pool. Policy state changes exhaustion semantics → ingest it. Reporting lag → settling window. Cost centers are GHE Cloud only → both API-only levers unavailable on Business-without-enterprise.

Simulation mode & test architecture (MSW)

No live credentials exist yet, so simulation mode is the primary development and verification surface — and it ships as an offline/demo mode admins can toggle. A single MSW (Mock Service Worker) layer fakes the GitHub billing/budget API and is the shared source of truth for three consumers, so behaviour and tests can't drift.

  • One mock, three consumers: simulation mode · Playwright e2e · unit/component tests.
  • Models: pooled draw, two-phase routing, ULB precedence, lowest-headroom-wins, the non-settable included-usage cap, the API-only CCULB, pagination, and the mutations (POST/PATCH/DELETE) so simulate-before-apply and both rebalancers run end to end. Edge fixtures: ULB display-bug, $0 block, cap-bound team, promo→standard cliff.
  • Stateless / deterministic: reset to fixtures each run; e2e asserts on the request issued + immediate response, not cross-request persistence; apply→result within a session uses optimistic UI.
  • Safety: simulated actions never reach real GitHub or mutate a real control; the mode is unmistakable in the UI (persistent banner).
Verification gate — every change

Playwright e2e (headless, MSW-backed, deterministic) is the automated gate; Chrome MCP verification of the actual running app is the interactive confirmation. A change is done only when both are green.

Stack note

Assumes the local-first Electron/TypeScript desktop delivery; MSW/Playwright/Chrome MCP are TS-stack tooling. The architecture & sync sections above predate that pivot — reconcile separately.

Build phases

  1. Simulation harness + scaffold (first) — monorepo, packages, Electron shell, ApiClient, Drizzle schema, MSW + fixtures, simulation toggle, and the Playwright + Chrome-MCP gate. Smoke test: "Sync now" pulls a report from MSW into SQLite, verified through the gate.
  2. Read-only observability — ingest, snapshots, dashboards.
  3. Full control administration — API-first CRUD of all controls incl. the two API-only levers; ULB-bug repair.
  4. Forecasting — run-rate, cliff, runway, P50/P90, backtest.
  5. Auto-balancing, dry-run — both rebalancers, simulate only.
  6. Guardrailed auto-apply + grant lifecycle + custom alerting.
  7. Chargeback + audit export.

Every phase passes the verification gate (Playwright + Chrome MCP against MSW) before it's done.

Part 5 — Open questions before build

Five things to settle first

  1. Enterprise or Business, and github.com or GHE.com? Gates cost centers, CCULBs, included-usage caps, allowances, API host.
  2. "AI credit paid usage" enabled? Determines whether the metered rebalancer is in scope.
  3. Cost centers in use, driven by enterprise teams aligned to DEWR division/branch?
  4. Pool posture: hard-stop at pool or controlled metered budget? Sets default prevent_further_usage.
  5. Auto-apply appetite: recommend-only or guardrailed auto-balancing — and the reserve buffer % and grant rollover policy (revert vs re-baseline)?
